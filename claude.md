# 【プロジェクト憲法】

## 1. 開発環境とアーキテクチャの原則
- **ローカル環境の保持**: 開発は Mac (OrbStack) + VS Code の Dev Containers（またはDockerコンテナ）内で行い、ホストOSの環境を絶対に汚さないでください。必要なツールはコンテナ内にインストールしてください。
- **インフラ管理**: インフラは原則として GCP を使用し、すべてのリソース構築は Terraform (`terraform/` ディレクトリ) で IaC化してください。コンソールからの手作業は極力排除します。
- **フルオートメーション**: スクリプトの作成、コマンドの実行、デプロイ作業など、Claude Codeで自動化できるものはすべてあなたが実行・提案してください。

## 2. 厳格な開発・デプロイ・コミットフロー（最重要）
コードを修正した際、**「ただ修正してすぐに git commit する」ことは絶対に禁止**します。必ず以下のループを守ってください。
1. **コード修正**: 実装を行う。
2. **テスト・デプロイ**: 修正後、すぐにビルドし、ローカルまたは検証環境（GCP等）へデプロイするスクリプトを実行する。
3. **実機動作確認**: デプロイされたエンドポイント（URL）やアプリケーションを叩き、エラーが出ないか、意図した動作になっているかを検証する。
4. **結果の判断**:
   - ✅ **問題なし**: ここで初めて `git add` と `git commit` を実行する。
   - ❌ **不具合あり**: エラーログを確認し、修正して [2] に戻る。
   - ⛔ **解決困難**: どうしようもなく沼にハマった場合は、元のコード（直前のコミット）に `git restore` 等でロールバックし、アプローチを再考する。

## 3. セキュリティと設定の一元管理
- **機密情報のハードコード禁止**: GCPのプロジェクトID、APIキー、パスワードなどの環境依存値や機密情報は、ソースコード（Terraform含む）に絶対にベタ書きしないでください。
- **.env による一元管理**: 設定値はプロジェクトルートの `.env` ファイルに集約し、各プログラム（Python, TS, Shell, Terraform等）はそこから環境変数を読み込む設計にしてください。
- **Gitへの漏洩防止**: `.env` 本体や、Terraformのステートファイル（`.tfstate`）、キーファイル（`.json`）は必ず `.gitignore` に含め、代わりに `.env.example` などのテンプレートファイルを作成してコミットしてください。

## 4. ドキュメントの同期
- コードの大きなアーキテクチャ変更や、新しい環境変数の追加、デプロイ手順の変更があった場合は、コミットする前に必ず `README.md` を最新の状態にアップデートしてください。第三者がクローンしてすぐに動かせる状態を維持してください。
