### 【導入】なぜ作ったのか

物件を見に行く前の下調べが、毎回だいたいこういう流れになる。

国土交通省の「不動産情報ライブラリ」を開いて取引価格を調べる。同じエリアで比較するためにタブを何枚か開いて、数字をコピーしてExcelに貼る。ハザードマップポータルを別タブで開いて、同じ住所を検索し直す。用途地域は都市計画のPDFを探してスクロールする。

「この物件、本当に適正価格なのか」を判断するための情報を揃えるだけで、1件あたり30〜60分。

プロの不動産エージェントでも、個人で物件を調べている人でも、この「調べる手間」は本質的に変わらない。2020年代に、公開データが整っている国で、なぜこれがまだ手作業なのか。

正直、この疑問が最初の動機だった。

「国土交通省のAPIは公開されている」「Google の生成AIは地域の専門知識を持っている」「Next.js と GCP なら一人でも本番に持っていける」——この3つが揃ったとき、「これは自動化できる」と確信した。

それが **物件目利きリサーチ**（[https://mekiki-research.com](https://mekiki-research.com)）の始まりだ。

### 【技術的アプローチ】どう作ったか

#### データソース: 国土交通省「不動産情報ライブラリ」API

中核となるのは国土交通省が無償公開している **不動産情報ライブラリ API（XIT001）** だ。市区町村コードと年を指定すると、実際の取引価格・坪単価・面積・用途などのレコードがJSONで返ってくる。CC BY 4.0 ライセンスなので、加工・再利用も自由。

問題は「市区町村コード」をどうやって取得するかだった。ユーザーが入力するのは地図上の座標（緯度・経度）であって、自治体コードではない。そこで国土地理院のリバースジオコーダ（GSI API）を使い、座標 → 住所 → 市区町村コードという変換パイプラインを実装した。

さらにハザード情報（洪水浸水深・土砂災害警戒）は同APIのタイルベースエンドポイント（XKT026, XKT029）から、用途地域・学区・医療機関・最寄り駅も対応エンドポイントで並列取得している。

#### キャッシュ: GCS + Cloud Run

MLIT APIはレスポンスが遅く、同じエリアへの重複リクエストも多い。そのためAPIレスポンスを **Google Cloud Storage（GCS）にJSONでキャッシュ**する仕組みを入れた。キャッシュHITならGeminiの生成も不要でミリ秒で返る。

バックエンドは **Hono + Node.js** を **Cloud Run** にデプロイ。TypeScriptで型安全に書き、Zodで入力バリデーション。コールドスタートの遅延は最初のリクエストで1回だけ我慢してもらう構成にしている（Min Instances = 0 でコスト優先）。

#### AI: Gemini 2.5 Flash + Imagen 4

取引データ・ハザード情報・生活環境情報が揃ったところで、**Gemini 2.5 Flash** に渡して10項目の専門家レポートを生成する。プロンプトは「20年以上のキャリアを持つ不動産鑑定士兼エージェント」というペルソナを設定し、数字の羅列でなく「目利きの見立て」として読めるテキストを引き出している。

セクションは以下の10項目：
1. エリア総評
2. 子育て・生活環境スコア
3. 歴史・地形の特徴
4. 開発・再開発動向
5. 活用できる補助金・助成金
6. 直近のニュース・トピックス
7. エリアの将来予想
8. 人口の増減予想
9. リアルな住環境と注意点
10. 不動産プロの視点：物件の魅力とおすすめポイント

「暮らしのイメージ画像」の生成は2段階アーキテクチャにした。まず Gemini にエリア特性（雪国・都市部・海沿いなど）を踏まえた英語プロンプトを動的生成させ、それを **Imagen 4 Fast** に渡す。静的なプロンプトを使うと全エリアで同じような郊外の家の絵になってしまうという問題をこれで解消した。

#### フロントエンド: Next.js 15 + Firebase

フロントエンドは **Next.js 15 App Router** で構築。認証は **Firebase Authentication**（Google OAuth）、ユーザープランと検索履歴の保存は **Firestore**。

ペイウォールは3層構造：
- ゲスト（未ログイン）: 1日1回・基本情報のみ
- Free: 1日3回・レポート3項目まで
- Pro: 無制限・全機能・PDF出力

PDF出力は **dom-to-image-more + jsPDF** で実装。地図（Leaflet）・暮らしイメージ・グラフ（Recharts）を含む複雑なDOM構造をキャプチャして、A4縦のPDFに落とし込んでいる。Leaftletのタイル画像がCORSでブロックされる問題は、国土地理院タイルをproxy設定なしでキャプチャするワークアラウンドで解決した。

#### インフラ: GCP + Terraform

インフラはすべて **Terraform** でIaC化している。Cloud Run（フロント・バック）、Artifact Registry、GCS、IAMの構成をコードで管理。`terraform apply` で再現可能な状態を維持している。

#### セキュリティ

本番公開前にセキュリティ監査を自前で実施した。主な対応：

- **Stripe checkout の認証**: `uid` をリクエストボディで受け取っていた（任意のユーザーの代わりに課金セッションを作れる）バグを修正。Firebase ID Tokenのヘッダー検証に変更。
- **Firestore Security Rules**: 初期状態でルールが未設定だったため、`users/{uid}` への書き込みを本人のみに制限し、`plan` フィールドはクライアントから変更不可にするルールをデプロイ。
- **PostHog webhook**: シークレット未設定時のサイレントスキップを廃止し、503を返す設計に変更。

### 【今後の展望】どんな価値を届けたいか

現在はβ版で、対象は日本全国の住所・座標。まず「不動産調査の手間を減らす道具」として使ってもらうことを最優先にしている。

中期的に取り組みたいのは以下の3点。

**① 物件単位への精緻化**
現状は市区町村単位のデータだが、将来的には個別物件（番地・号室）レベルの分析に踏み込みたい。容積率・建ぺい率・日照計算・隣接地の状況など、プロが現地で確認していることをデータで補完する。

**② 不動産営業向けの業務ツール化**
1案件あたりの調査時間を大幅に短縮できる可能性がある。顧客へのプレゼン資料として直接使えるPDFレポートの品質向上、CRMとの連携など、実務フローへの統合を目指したい。

**③ 継続的なエリアウォッチ**
「このエリアの相場が3ヶ月前と比べてどう変わったか」を自動で通知する機能。物件購入を検討しているユーザーが、タイミングを見極めるための判断材料として使える。

最終的に目指しているのは、「プロの目利き」をデジタル化して、誰でもフェアに使えるようにすることだ。不動産の意思決定は人生で最も大きな金融的決断のひとつ。その判断を支える情報が、まだ不均衡に分配されている。その非対称を少しでも埋めたい。
